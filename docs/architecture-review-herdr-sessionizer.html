1. Shared project discovery module

✓ COMPLETED

Files involved

  • src/sessionizer.ts — lines 22-30, 34-36
  • src/worktree.ts — lines 186-198 (duplicate)
  • src/config.ts — expandHome() line 199
  • src/layouts/project.ts — shellQuote() line 227

Recommendation strength

Strong — verbatim duplication, clear extraction point, immediate locality gain.

Problem

listProjects() is defined identically in both sessionizer.ts and worktree.ts — 10 lines each, verbatim copies. sanitizeName() is also duplicated. expandHome() lives in config.ts but is conceptually a string utility. shellQuote() lives in layouts/project.ts but is a general-purpose shell quoting function. These are shallow modules: their interfaces are as complex as their implementations, and they're scattered across files with no locality. Any change (e.g., adding a new project root pattern) requires touching multiple files.

Solution

Extract a src/discovery.ts module with a small interface: listProjects(roots): string[], sanitizeName(name): string, expandHome(path): string, normalizePath(path): string, shellQuote(value): string. Both sessionizer and worktree import from it. The interface hides filesystem enumeration details behind a single function call.

Benefits

  • Locality — all project discovery logic in one place. Change once, fix both callers.
  • Leverage — callers get a 5-line call instead of duplicating 10-line functions.
  • Testability — listProjects() becomes testable without importing sessionizer or worktree.
  • Deletion test: deleting the module would scatter the logic back across callers — it earns its keep.

2. Consolidate worktree existence resolution

✓ COMPLETED

Files involved

  • src/worktree.ts — lines 70-144 (core logic)
  • src/ops/worktrees.ts — thin CLI wrapper
  • src/client/errors.ts — HerdrError used for parsing

Recommendation strength

Completed — the worktree recovery path now lives behind WorktreeResolver and has focused parser and flow coverage.

Problem

The worktree flow in worktree.ts has two parallel paths (open existing → bootstrap, or create → catch → recover) that share the same goal but are implemented as separate branches. The recovery from "already exists" errors involves parsing stderr strings, running git worktree list --porcelain, and matching by branch name or slug — all inline. This is untested and tightly coupled to the module's main flow. The asHerdrError() helper tries to coerce any error into a HerdrError, which is fragile.

⚠ Contradicts ADR-0001 (implicit): The current design treats worktree existence as an error path. A cleaner design would separate "check existence" from "create or open" — making the happy path simpler and the error path explicit.

Solution

Extract a WorktreeResolver module with a single interface: resolveExisting(project, branch): ExistingWorktree | null. It encapsulates all 4 resolution strategies (stderr parsing, git porcelain by branch, git porcelain by slug, and future strategies). The worktree flow becomes:
1. try resolveExisting → if found, open and optionally bootstrap
2. else try create → if HerdrError, resolveExisting again → open
No error-path-as-control-flow pattern.

Benefits

  • Testability — resolution strategies become unit-testable in isolation.
  • Locality — all "find an existing worktree" logic lives behind a single seam.
  • Leverage — callers get a simple resolveExisting → open or create flow.
  • Deletion test: deleting the module would scatter git-parsing and stderr-parsing back into worktree.ts.

3. Deepen the layout builder seam

✓ COMPLETED

Files involved

  • src/layouts/project.ts — 274 lines (the largest module)
  • src/ops/panes.ts, src/ops/tabs.ts — called extensively
  • src/config.ts — tab/pane config types

Recommendation strength

Completed — command building and error policy now have dedicated seams, and the layout orchestrator has focused coverage.

Problem

layouts/project.ts is the deepest module at 274 lines, but its interface is large (takes workspace, cwd, config, tabs, panes instances, plus optional context). It exposes 4 internal functions (configureExistingTab, createAndConfigureTab, configureTabPanes, configurePane) that are only used internally but aren't private (they're exported at module level). Implicit state: nextPaneIndex is threaded through as a mutable counter. Silent error swallowing: .catch(noop) appears 6 times — every herdr call can fail silently. Pane ID generation: workspacePaneId() parses/reconstructs IDs via string manipulation.

Solution

Split into two modules behind a single seam:
a) LayoutOrchestrator — public interface: createProjectLayout(workspace, cwd, config, options?). Hides the tabs/panes ops instances and the iteration machinery.
b) LayoutErrorPolicy — explicit error handling strategy: retry, skip, or abort per operation. The current .catch(noop) becomes one strategy.
c) Move buildPaneCommand, interpolateCommand, applyCommandContext into a CommandBuilder module — they're pure functions with no herdr dependency.

Benefits

  • Leverage — callers get createProjectLayout(ws, cwd, cfg, opts?) and nothing else.
  • Locality — pane ID generation, command building, and error policy each get one place.
  • Testability — CommandBuilder and LayoutErrorPolicy are testable without herdr.
  • Deletion test: deleting LayoutErrorPolicy would scatter error handling; deleting CommandBuilder would inline command construction.

4. Unify sessionizer and worktree workflows

SPECULATIVE

Files involved

  • src/sessionizer.ts — 53 lines
  • src/worktree.ts — ~200 lines
  • src/layouts/project.ts — shared

Recommendation strength

Speculative — the workflows share listProjects + layout application, but are genuinely different in branching logic. Extracting a common base risks a leaky abstraction.

Problem

Both sessionizer.ts and worktree.ts follow the same high-level pattern: load config → list projects → pick → create/open workspace → apply layout → focus. But worktree.ts has 4× the complexity due to branch handling and error recovery. Extracting a common base WorkspaceWorkflow would abstract the shared steps but would need hooks for the divergent parts (branch prompt, worktree resolution). This is speculative because the abstraction boundary may not be worth the indirection.

Solution

Only extract if candidates 1 and 2 are done and duplication remains. The pattern would be a WorkspaceWorkflow base class or function with strategy hooks for the divergent steps. However, candidate 1 already eliminates the main duplication (listProjects). The remaining divergence is genuine business logic, not accidental duplication.

Benefits

  • Would reduce boilerplate in both flows.
  • But risks a leaky abstraction — the "pick → create/open → layout" pattern is simple enough that two copies are clearer than one parameterized abstraction.

5. Ops wrappers — keep or deepen?

SPECULATIVE

Files involved

  • src/ops/panes.ts — 30 lines
  • src/ops/tabs.ts — 32 lines
  • src/ops/workspaces.ts — 39 lines
  • src/ops/worktrees.ts — 82 lines
  • src/client/herdr.ts — 37 lines

Recommendation strength

Speculative — these modules are shallow by nature (CLI wrappers) but pass the deletion test. Only Worktrees has enough logic to warrant deepening.

Problem

The four ops modules are essentially one-to-one translations from TypeScript method calls to herdr CLI arguments. They're shallow — the interface is nearly as complex as the implementation. However, they do provide type safety and JSON response parsing. The deletion test is interesting: deleting them would scatter CLI argument construction and JSON response parsing across callers, so they do earn their keep.

Observation

src/ops/worktrees.ts is the thickest at 82 lines. Its create() and open() methods have complex argument building with multiple optional parameters. This is the one ops module that could benefit from a builder pattern or option-object refinement.

Recommendation

Keep the ops wrappers as-is — they're shallow by necessity and pass the deletion test. Only consider deepening Worktrees if candidate 2 (WorktreeResolver) is implemented and reveals further complexity in the ops wrapper.
